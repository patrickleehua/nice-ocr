## Why

「全部结果」和「审核工作台」当前的作用域是**隐式且两页不对称**的：作用域只能靠 URL 是否携带 `?batchId=` 来决定，用户无法在页内选择或切换批次。

- 全部结果：无 `batchId` 时默认「全部批次」（合理），但页内只能「清除」批次过滤，不能选择其他批次；要切批次必须退回批次列表重新进入。
- 审核工作台：无 `batchId` 时**默认落到最近一个批次**（`batches[0]`），既不是「全部」也不是用户主动选择的，是系统替用户猜的；且结构上只支持单批次，无法跨批次审核。
- 隐藏隐患：结果表/审核台的字段列取自**全局活动场景**（`/api/fields` → `getActiveScenarioId()`），而批次各自带 `scenarioId`。一旦批次间场景不同，「全部」混批视图的列就会错位。

目标：让两页拥有一致、显式、可就地切换的「批次作用域」——**默认全部，可隔离到任一批次**，并顺手修复场景错位。

## What Changes

- 在「全部结果」与「审核工作台」加入统一的**批次作用域选择器**（全部 / 批次A / 批次B…），默认「全部」。
- 作用域以 URL `?batchId=` 为单一事实源，并在 结果 ↔ 审核 之间互相保留，切换标签不丢上下文。
- **审核工作台支持跨批次工作流**：作用域为「全部」时，左侧文件列表混排所有批次的待审文档并标注所属批次（待办收件箱式）；选中批次则回到单批次隔离视图。
- **隔离 = 进入批次工作区上下文**：选中某批次后，结果/审核仅展示该批次数据，字段列采用该批次场景，导出默认带该批次模板，并显示 `BatchWorkspaceNav`。
- **混场景处理**：
  - 结果页「全部」扁平表按结果集中出现的场景数自适应——单场景出完整列；多场景退化到公共核心列并提示。
  - 审核台明细按所选文档所属批次场景出列（永远单场景，无错位）。
- 审核台默认行为由「最近批次」改为「全部」跨批次待办视图。

## Capabilities

### New Capabilities
- `batch-scoped-review-results`: 在结果与审核两页提供统一的批次作用域（默认全部，可隔离到单批次），含跨批次审核待办与场景感知出列。

### Modified Capabilities

None.

## Impact

- 前端：`results-page`、`review-page` 引入作用域选择器与作用域状态；新增 `BatchScopeSelect` 共享组件；`BatchWorkspaceNav` 复用不变。
- 字段解析：`/api/fields` 支持按 `scenarioId`/`batchId` 返回指定场景字段（不改全局活动场景）；`useFieldSchema` 接受作用域参数。
- 审核数据通路：新增跨批次文档列表能力（按审核状态/名称过滤、可定位单批次），供审核台「全部」模式左侧列表使用。
- 结果数据通路：`/api/rows` 响应补充当前结果集涉及的场景集合，驱动混场景列退化。
- 测试：作用域默认/切换/隔离、跨批次待办列表、混场景列退化、按文档场景出列。
